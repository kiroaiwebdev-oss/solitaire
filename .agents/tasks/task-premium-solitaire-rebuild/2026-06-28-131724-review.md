# Premium Solitaire Rebuild

A complete rewrite of a Klondike Solitaire web game from basic to premium quality: new core engine with unlimited undo/redo, hint system, auto-complete, 4 difficulty levels, 3 scoring modes; new canvas-based UI layer with neomorphism styling, particle system, animation manager, and responsive layout; 5 gaming portal adapters; PWA support; 755 tests passing across 3 suites. The architecture uses a clean separation between game logic, UI, platform, and system layers — all vanilla JS with zero dependencies, all assets synthesized at runtime.

**Watch for:** Keyboard shortcuts are completely non-functional due to event name mismatch between Input and App (confirmed). Double-tap auto-move passes coordinates where a Card object is expected, so it never succeeds (confirmed). Achievements, daily challenge, and progression screens always display empty/default data because the App never pushes state to Screens (confirmed).

## High-level view

The game engine uses full state snapshots for undo/redo (capped at 500), and the hint system scans all valid moves across waste, tableau, and foundation. The undo implementation has a double-penalty quirk: it restores the pre-move score and then subtracts another 20 points, meaning you lose both the points you earned and a penalty fee.

The input system supports multi-touch, double-tap, long-press, swipe, and keyboard shortcuts — but the keyboard event bus disconnects from the app layer. Input emits `'shortcut'` events; the App listens for `'keyboard'` events that never fire. All keyboard shortcuts (Ctrl+Z, H, N, A, Escape, Space) are dead.

Several data-flow paths from systems to screens are missing: `setDailyData`, `setAchievements`, and `setProgression` exist on Screens but are never called. The daily challenge calendar, achievements gallery, and progression indicator on menus always show zeroes.

The responsive layout dynamically recomputes card sizes and adapts stack offsets to prevent overflow — the one aspect of "premium quality" that consistently works across viewports. Settings controls all produce real state changes that persist.

The test suite at 755 tests covers game logic, module structure, and platform adapters — but never exercises the App class wiring. Every bug in this review exists at the integration boundary between modules that are individually well-tested.

## Keyboard shortcut wiring gap

Input's `_onKeyDown` method emits events named `'shortcut'`:

```javascript
this._emit('shortcut', { action: 'undo' });
this._emit('shortcut', { action: 'redo' });
this._emit('shortcut', { action: 'hint' });
```

But the App registers:

```javascript
this.input.on('keyboard', (e) => this._onKeyboard(e));
```

No code path ever emits a `'keyboard'` event. Every keyboard shortcut — undo, redo, hint, new game, auto-complete, pause, draw from stock — is non-functional. On a desktop browser, the only way to interact is mouse/touch.

## Double-tap auto-move passes wrong arguments

`_onDoubleTap` in main.js:

```javascript
this.game.autoMoveToFoundation(coords.x, coords.y, this.layout);
```

`Game.autoMoveToFoundation` signature:

```javascript
autoMoveToFoundation(card) {
  const tableauLoc = this.tableau.findCard(card);
  ...
}
```

The method calls `this.tableau.findCard(card)` which does `this.columns[c].indexOf(card)`. Since `card` is actually a number (the x coordinate), `indexOf` will never find it, and the method returns false every time. The fix needs to resolve the tap coordinates to the card at that position before calling `autoMoveToFoundation`.

## Screens data pipeline is disconnected

The Screens class provides setters:
- `setDailyData(data)` — populates streak, calendar, completed status
- `setAchievements(data)` — populates unlocked list and progress
- `setProgression(data)` — populates level and XP

None are called anywhere in main.js. The daily challenge screen always shows streak 0, empty calendar, and an enabled "Start Today's Challenge" button (since `dailyData.completed` is always false — even after completing today's challenge). The achievements screen shows 0/26 unlocked. The main menu's level indicator never appears (default level is 1, and rendering is gated on `level > 1`).

After `_loadAllState()` and after each `_saveAllState()`, the app should push updated data to screens.

## Difficulty and scoring mode selection is vestigial

The Game class defines `DIFFICULTY.EASY/MEDIUM/HARD/EXPERT` and `SCORING_MODE.STANDARD/VEGAS/NONE`. The mode select screen offers four buttons labeled with these modes. But `_startNewGame` constructs the Game with only `drawCount`, `hardMode` boolean, and `hardModeTime`. It never passes `difficulty` or `scoringMode`.

The practical effect: Vegas scoring (-$52 start, $5 per foundation card) is never activated. The `timerCountsDown` path (which uses `_isTimed()` → checks difficulty) is always false. Timers work only through the legacy `hardMode + hardModeTime` path. The user sees "Medium (Draw 1, Timed)" on the button but the game uses a different code path than the one designed for it.

## Statistics chart is decorative

In `_renderStatistics`, the bar chart uses:

```javascript
const bh = chartH * 0.6 * (0.4 + Math.random() * 0.6);
```

Bar heights are random on every render frame (the chart visually jitters every frame). The color logic uses `i < s.won` which just colors the first N bars green regardless of actual game outcomes. Real per-game history exists in `saveManager.loadGameHistory()` but is never wired into the chart.

## cardCascade uses setTimeout for stagger

The animation manager's `cardCascade` uses `setTimeout` for stagger timing rather than accumulating delay within the tween system. If the animation manager is cleared mid-cascade (e.g., screen transition), the delayed callbacks will still fire and add orphaned tweens to the system. Not a crash risk, but can produce unexpected visual artifacts.
