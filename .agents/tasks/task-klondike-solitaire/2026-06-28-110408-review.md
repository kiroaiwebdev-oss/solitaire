# Greenfield Klondike Solitaire HTML5 game

A complete Klondike Solitaire game built from scratch in vanilla ES modules with Canvas2D rendering and Web Audio synthesis. The architecture separates core engine (loop, input, render, audio, math), game logic (card, deck, tableau, foundation, stock, drag, game orchestrator), UI (HUD, screens), progression systems (XP, daily challenges, achievements, save), and platform adapters (5 portals + build tool). Total footprint is ~7,700 lines across 33 source files, zero external dependencies, and all 627 test assertions pass across three test suites (logic, DOM/module, SDK probes).

**Watch for:** Input coordinate scaling uses physical pixels but game logic uses logical coordinates, creating a 2x mismatch on high-DPI displays that breaks all hit-testing (confirmed). The `_drawFelt` texture function draws individual 1px rects across the entire canvas every frame -- a performance concern on mobile (confirmed). The `_returnCards` method on failed drops does not call `snapBack()`, so cards visually teleport instead of animating back (confirmed). The "Continue" button on the main menu is permanently disabled because game state is never saved on unload (confirmed).

## High-level view

Input handling converts pointer events into canvas coordinates using `(clientX - rect.left) * (canvas.width / rect.width)`. Since the renderer sets `canvas.width = rect.width * devicePixelRatio` and then calls `ctx.scale(dpr, dpr)`, the coordinate math produces physical-pixel values while the game logic operates in logical pixels. This DPR mismatch means every click lands at 2x the intended position on Retina screens -- the game is unplayable on any non-1x display.

The drag system tracks card sources, follows the pointer, and validates drops against correct Klondike rules. The snap-back path on invalid drops does not animate -- `_returnCards` clears the dragging flag without calling `DragSystem.snapBack()`, so cards teleport to their recomputed position.

Platform adapters follow a consistent interface with graceful SDK-absent fallback. All adapters complete init without their SDK, report adblocked status, and show a countdown overlay instead of real ads. The build system generates per-platform zips with SDK injection and cache-busting using only Node stdlib.

The progression/daily/achievement subsystems use localStorage with deterministic seeded shuffles for daily challenges. Streak logic handles the today/yesterday boundary but does not account for timezone crossings during a streak (possible concern -- would need to check player impact).

<summary>Issues (7)</summary>

1. **DPR coordinate mismatch** (confirmed) -- `Input._getCanvasCoords` multiplies by `canvas.width / rect.width` which equals `devicePixelRatio`, but all game logic uses logical coordinates. Remove the DPR multiplication so pointer coords match rendered positions.
2. **Felt texture per-frame cost** (confirmed) -- `_drawFelt` iterates every 4th pixel across the viewport filling 1px rects each frame (~16,800 fillRect calls at 960x540). Replace with a pre-rendered offscreen canvas blitted once per frame.
3. **Snap-back does not animate** (confirmed) -- `_returnCards` in `App` sets `card.dragging = false` but never calls `DragSystem.snapBack()`. Cards teleport on invalid drops.
4. **Renderer scale accumulation on DPR change** (likely) -- `_onResize` calls `ctx.scale(dpr, dpr)` after resetting canvas dimensions. Safe for same-DPR resizes, but dragging a window between monitors with different DPRs leaves a stale transform until the next resize event. Use `ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` for safety.
5. **Undo state is single-depth only** (confirmed) -- `saveUndoState` overwrites the previous snapshot. A second move makes the first irreversible. Document as intentional or switch to a stack.
6. **No game-state save on page unload** (confirmed) -- `SaveManager` has `saveGameState`/`loadGameState` methods but `App` never calls them. Mid-game progress is lost on tab close, and the "Continue" button is permanently disabled.
7. **Spurious reveal-card score bonus** (confirmed) -- `_executeDrop` awards `SCORING.REVEAL_CARD` when the source column's new top card is face-up, even if it was already face-up before the move. Should only award when a previously face-down card was flipped.

## DPR coordinate scaling bug

The `Input` class computes canvas coordinates as:

```js
_getCanvasCoords(e) {
  const rect = this.canvas.getBoundingClientRect();
  const scaleX = this.canvas.width / rect.width;
  const scaleY = this.canvas.height / rect.height;
  return {
    x: (e.clientX - rect.left) * scaleX,
    y: (e.clientY - rect.top) * scaleY
  };
}
```

Since `Renderer._onResize` sets `this.canvas.width = rect.width * dpr`, the `scaleX` factor equals `dpr`. All card positions, HUD button rects, and screen button coordinates are computed in logical pixels (the renderer's `logicalWidth` = `canvas.width / dpr`). So a click at CSS-pixel (100, 200) on a 2x display produces coords (200, 400), but the card at that position has `card.x = 100`. Every interactive element is unreachable on non-1x screens.

The fix: since the renderer applies `ctx.scale(dpr, dpr)`, CSS-pixel coordinates already correspond to drawing space. `_getCanvasCoords` should return `(clientX - rect.left, clientY - rect.top)` directly.

## Snap-back teleportation

In `_onPointerUp`, when a drop fails:

```js
if (!moved) {
  this._returnCards(source, cards);
  if (this.audio) this.audio.play('error');
}
this._positionCards();
```

`_returnCards` only sets `card.dragging = false`. Then `_positionCards()` overwrites `card.x/y` directly. The `DragSystem` stores `returnPositions` and has a `snapBack()` method that calls `card.animateTo()` for smooth motion, but it's never invoked.

## Per-frame felt texture rendering

```js
_drawFelt(ctx) {
  const w = this.renderer.logicalWidth;
  const h = this.renderer.logicalHeight;
  ctx.fillStyle = 'rgba(0,0,0,0.02)';
  for (let gy = 0; gy < h; gy += 4) {
    for (let gx = 0; gx < w; gx += 4) {
      if ((gx + gy) % 8 === 0) {
        ctx.fillRect(gx, gy, 1, 1);
      }
    }
  }
}
```

On a 960x540 logical viewport, this produces ~16,800 fillRect calls per frame. The pattern is static and should be rendered once to an offscreen canvas, then blitted as a single `drawImage` call each frame.

## Missing game-state persistence

`SaveManager` defines `saveGameState`, `loadGameState`, `hasSavedGame`, and `clearGameState`. The `Screens` class has `setHasSavedGame(val)` and the "Continue" button checks `this.hasSavedGame`. But `App` never serializes game state on pause/close and never calls `this.screens.setHasSavedGame(true)`. Adding a `beforeunload` listener or saving on pause/menu would complete this feature.

## Spurious reveal-card score bonus

In `_executeDrop`, after a tableau-to-tableau move:

```js
if (source.type === 'tableau') {
  const col = this.game.tableau.columns[source.index];
  if (col.length > 0 && col[col.length - 1].faceUp) {
    this.game.score += SCORING.REVEAL_CARD;
  }
}
```

This awards the bonus whenever the new top card is face-up, even if it was already face-up before the move (player moved only part of a face-up sequence). The fix: record whether the top card was face-down before `removeSequence`, then award only if it flipped.

## Auto-complete premature trigger

`canAutoComplete()` checks that stock is empty and all tableau cards are face up, but ignores remaining waste cards. If waste cards exist that can't reach foundations, `autoCompleteStep` will try one tick, find no valid move, and disable itself -- it terminates correctly, but the button appearing and then immediately disappearing is confusing UX.

## Test coverage gaps

Not tested: input-to-game-state integration (the DPR bug would be caught here), the auto-complete loop under edge conditions, screen transitions, and win animation physics. The SDK probes correctly prioritize testing adapters without their SDKs.
